# 设计空间扫描与可行性边界 — T4

## 目的

交接文档 §4 的 T4：扫往返窗口 D 与注错情形，定位 ρ=D/H 的"划算/不划算"交叉边界，
并演示"请求窗口外被拒"（验证 T5 的 C2 时间窗条件）。

## 1. D 扫描（模型 + 仿真锚点）

行级(A)的额外内存 = D·L、恢复时延 = D·T_l 随 D 线性增长；整帧重传(B)固定为整帧(H)。
故 A 的优势恰为 **H/D**，**划算边界 = ρ = D/H**：D<H 行级严格更优，D→H 退化为 B。

模型（`python tools/recapture_model.py --width 4 --height 8 --bpp 1 --rt-lines 1`，H=8）：

| D(行) | ρ=D/H | A 额外内存 | 省内存/时延 | 划算? |
|---:|---:|---:|---:|:--:|
| 1 | 0.125 | 4 B | 8× | 是 |
| 2 | 0.250 | 8 B | 4× | 是 |
| 4 | 0.500 | 16 B | 2× | 是 |
| 8 | 1.000 | 32 B | 1× | 退化=B |

**仿真锚点**（`tb_recapture_strategy_compare` 的 runner `tb_strat_a_d1/d2/d4`，重采集在坏行后
D 行发出）：实测 `buffer_lines = D`（上游缓存随 D 增长），且 **A 的恢复流量恒为 1 行（18 B）**，
与 D 无关——印证"行级只重发坏行"。三个 D 均 `recovered=1`（D<H，帧内可恢复）。

```
tb_strat_a_d1: buffer_lines=1 recovery_bytes=18 recovered=1
tb_strat_a_d2: buffer_lines=2 recovery_bytes=18 recovered=1
tb_strat_a_d4: buffer_lines=4 recovery_bytes=18 recovered=1
```

> 注：恢复时延的**绝对值**在本机仿真受 AXI 异步 FIFO 排空动态影响、在 4 像素小帧上不随 D 单调，
> 故"时延 ∝ D"取自模型（T1 给出恢复时延 = D·T_l）；仿真侧给出的是干净的缓存/流量标度。

## 2. 时间窗边界 C2（请求窗口外被拒）

`retry_request_ctrl` 只保留**一个未决请求**（锁存最近一次定位的错误）。若行错误到达
快于被服务/重采的速度，较早行的请求会被覆盖、该行**不被恢复**——这是可行性条件 C2
（请求须在下一个之前被服务，否则落在可恢复窗口之外）的仿真对应。

`tb_recapture_window_limit`：一帧内 line3 与 line5 连续出错、其间不重采，再发一次重采集：

```
PASS: tb_recapture_window_limit  single outstanding request:
      line5 recovered (slot5 clean), line3 lost (slot3 empty), context=6
```

- 最近错误（line5，帧内 1 基=6）占据单一未决槽并被恢复；
- 较早错误（line3）请求被覆盖 → slot3 永久为空（丢失）。

**结论**：行级重采集的"服务能力"是有限的——在 C2 意义下，注错密度必须低于服务速率
（单未决请求时即"两次错误间须完成一次重采"），否则退化为部分丢失。这给出了 ρ 划算区之外
的第二条边界：**注错密度边界**。

## 3. 两条边界合起来（设计空间结论）

行级选择性重采集的"划算可行域"由两条边界界定：
1. **ρ = D/H ≪ 1**（往返窗口 ≪ 帧高）——内存/时延优势的来源；
2. **注错密度 < 服务速率**（C2）——单未决请求下，错误须被及时服务。

二者同时满足时，行级以 H/D 倍更小的内存/时延恢复坏行，且不像"只丢"那样永久丢失。
这正是 §3 命题"在某条件下行级严格优于整帧重传/整帧丢弃"的完整设计空间刻画。
