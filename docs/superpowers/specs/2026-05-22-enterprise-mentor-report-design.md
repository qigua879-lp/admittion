# 企业导师版报告重写设计说明

## 1. 目标

新文档命名为 `MIPI CSI-2数字逻辑RX端设计及仿真验证说明.md`，面向企业导师阅读，用学生身份说明我在本阶段完成了哪些数字逻辑 `RX` 端设计工作、我是如何做仿真验证的、当前结果能够支撑到什么程度，以及哪些内容仍需要后续完善。

这份文档不是论文正文，也不是工程 signoff 报告，而是一份适合阶段性交付的技术说明。

## 2. 受众与语气

- 主要受众：企业导师
- 次要用途：作为实习阶段交付材料留档
- 人称：统一使用“我”
- 语气：正式但不过度学术，保留学生身份，不写成项目负责人总结
- 术语策略：尽量通俗表达；“背压”统一改成“写入端阻塞”或“下游来不及接收”

## 3. 文档结构

新文档采用“论文式结构 + 企业交付导向”的混合写法，结构如下：

1. 简要说明  
只简短说明我完成的是 `MIPI CSI-2` 数字逻辑 `RX` 端设计与验证，不重复导师已经知道的项目背景。

2. 预期内容与当前完成情况对比  
用表格对照“原本预期完成内容”和“当前已完成情况”，作为全文总览。

3. 主要完成内容说明  
按功能模块说明已完成内容，包括：
- 多 lane 数字输入接收与整理
- `CSI-2` 协议解析
- `ECC / CRC` 检查
- 帧行同步
- 像素重组
- 恢复机制
- 跨时钟缓存
- 写接口
- 基础预处理

4. 接口说明  
先给一张顶层接口总表，再给五张分表：
- 时钟与复位
- 输入接口
- 配置接口
- 写接口
- 调试与状态输出

5. 仿真与验证方法  
按验证层级说明方法：
- 模块级测试
- 系统级 wrapper 测试
- 异常注入测试
- 参数对比测试
- 正式波形截图留痕

6. 仿真验证结果  
直接插入关键结果表格和正式波形图，不只做文字概述。该部分按“先波形、后表格”的顺序组织，保证企业导师先看到验证过程，再看到结果汇总。计划包含：
- 正式波形图组，使用当前已生成的正式波形截图，覆盖主要验证过程
- 一张总表，字段为：`tb 名称 / 验证模块 / 验证功能 / 验证结果`
- 多格式结果对比表
- `1 / 2 / 4 lane` 配置结果表
- 恢复链和写入端阻塞相关结果表
- 必要时补充 55 个 testbench 正式重验证结果与 7 张正式波形截图之间的对应说明

7. 当前不足与后续建议  
诚实说明当前边界，包括：
- 当前聚焦数字逻辑 `RX` 端
- `1 lane / 4 lane` 验证深度不如 `2 lane`
- 最终板级定版未完成
- 更长时间稳定性和更强覆盖仍需补充

8. 可交付材料说明  
列清楚本次可以交付到企业的内容，例如源码、接口说明、测试文件、结果表、正式波形图和 Vivado 相关材料。

- 新建文件，不覆盖现有总览文档
- 文件名：`MIPI CSI-2数字逻辑RX端设计及仿真验证说明.md`
- 文档应可单独阅读

## 6. 风险与边界

- 如果直接照抄现有验证总报告，容易继续保留“工程总结”口气，因此必须重新组织语句。
- 如果接口部分只保留分散说明，不做总表，会降低企业导师快速浏览效率。
- 如果结果部分只放链接不放表，会削弱文档的独立交付价值。
